# =============================================================================
#  1. 设计理念
# =============================================================================
#  本构建系统采用"最小化核心 + 独立配方 + 可插拔模块"架构。
#  - 主脚本 (main/main.sh) 只负责流程编排，不包含具体构建逻辑。
#  - 公共库 (lib/) 提供可复用的工具函数（日志、下载、解压）。
#  - 全局配置 (config.sh) 允许一次性设定交叉编译目标、安装前缀等环境变量。
#  - 包配方 (packages/<包名>/recipe.sh) 以独立脚本形式定义每个软件包的元数据
#    和构建步骤。
#
#  所有变量遵循"就近定义原则"：主脚本定义路径和状态变量，配置脚本定义环境变量，
#  配方定义包相关变量。变量名采用 UPPER_CASE 风格，环境变量通常大写。

# =============================================================================
#  2. 项目目录结构
# =============================================================================
#  tuxwine/
#  ├── main/
#  │   └── main.sh           # 入口，流程调度
#  ├── lib/
#  │   ├── common.sh         # 日志函数 (info, warn, error)
#  │   ├── fetch.sh          # 下载函数 (download)
#  │   └── unpack.sh         # 解压函数 (extract)
#  ├── config.sh             # 可选全局环境变量（交叉编译等）
#  ├── packages/
#  │   └── <包名>/
#  │       ├── recipe.sh     # 包配方（必选）
#  │       └── patches/      # 可选补丁文件
#  ├── downloads/            # 源码包缓存（自动管理）
#  ├── work/                 # 构建工作目录（自动管理）
#  └── output/               # 最终安装文件树（按包名存放）

# =============================================================================
#  3. 主脚本变量 (main/main.sh)
#     所有变量由主脚本定义或使用，用户不应直接修改这些值。
# =============================================================================

# ----- 3.1 路径变量 -----
# MAIN_DIR          主脚本所在目录（绝对路径），即 main/
# PROJECT_ROOT      项目根目录，即 tuxwine/
# PACKAGES_DIR      ${PROJECT_ROOT}/packages  存放所有包配方的目录
# DOWNLOAD_DIR      ${PROJECT_ROOT}/downloads 源码包下载缓存目录
# WORK_DIR          ${PROJECT_ROOT}/work      构建工作空间
# OUTPUT_DIR        ${PROJECT_ROOT}/output    最终产物收集目录

# ----- 3.2 内部状态变量 -----
# PKG_RECIPE_DIR    关联数组: 包名 -> 配方目录路径，由 scan_recipes() 填充
# BUILT_PKGS        关联数组: 包名 -> 任意非空值，标记本轮已构建成功的包

# ----- 3.3 主脚本内部使用的局部变量（函数内） -----
# pkg_name          当前正在处理的包名
# recipe_dir        当前包的配方目录
# recipe_file       当前包的配方文件完整路径
# src_basename      从 SRC_URI 提取的源码包文件名
# src_dir_name      解压后的源码目录名（由 SRC_DIR 决定，未指定则自动推导）
# pkg_work          当前包在 work/ 下的私有工作目录
# src_dir           源码解压后的完整路径
# dest_dir          临时安装根目录（即 DESTDIR）
# dl_path           下载的源码包存储路径

# =============================================================================
#  4. 全局配置文件变量 (config.sh)
#     位于项目根目录，可选。如果存在，主脚本会在加载公共库后 source 它。
#     用于设定整个构建系统的通用环境，所有配方均会继承这些变量。
# =============================================================================

# 变量名                      | 必填 | 说明
# TARGET_HOST                 推荐   交叉编译的目标三元组，例: aarch64-linux-gnu
# BUILD_HOST                  推荐   宿主机三元组，例: x86_64-linux-gnu
# PREFIX                      推荐   软件包的安装前缀，例: /data/.../imagefs/usr
# PKG_CONFIG_LIBDIR           可选   覆盖 pkg-config 的搜索路径
# PKG_CONFIG_SYSROOT_DIR      可选   pkg-config 的 sysroot 路径
# MAKE_INSTALL_PROGRAM        可选   传递给 make install 的额外参数，
#                                   例: INSTALL_PROGRAM="/usr/bin/install -c"
#
# 注意: 这些变量是简单的 shell 变量（非环境变量），配方中可直接使用，
# 也可在函数内通过 export 提升为环境变量。

# =============================================================================
#  5. 配方文件变量 (packages/<包名>/recipe.sh)
#     每个包的构建描述文件，以 recipe.sh 命名。主脚本会 source 该文件，
#     因此它必须是有效的 Bash 脚本片段。
# =============================================================================

# ----- 5.1 必填变量 -----
# VERSION           软件版本号，如 "1.5.0"
# SRC_URI           源码下载链接（支持 http/https），如 https://zlib.net/zlib-1.3.tar.gz

# ----- 5.2 可选变量（元数据） -----
# PKGNAME           包名，默认使用配方所在文件夹名，用于工作目录和输出目录命名
# SRC_DIR           解压后源码目录名。若不设置，系统会从 SRC_URI 文件名去掉后缀猜测。
#                   当压缩包内目录名与文件名不一致时务必设置。
# BUILD_DEPENDS     (当前未使用) 未来用于指定宿主机需安装的构建依赖
# DEPENDS           (当前未使用) 未来用于指定必须先编译的其他包名

# ----- 5.3 配方内可访问的全局变量 -----
# 以下变量已在配方被加载时存在，可直接在 build(), install() 等函数中使用：
# DESTDIR           临时安装根目录，install() 必须安装到 $DESTDIR 下
# PROJECT_ROOT      项目根路径，可用于引用补丁等资源
# PREFIX            由 config.sh 提供，建议的安装前缀
# TARGET_HOST       交叉编译目标三元组
# BUILD_HOST        宿主机三元组
# MAKE_INSTALL_PROGRAM  传递给 make install 的额外选项

# ----- 5.4 配方必须定义的函数 -----
# build()           执行配置与编译。在解压源码并执行 prepare() 后被调用。
# install()         将编译产物安装到 $DESTDIR 下，绝不能安装到系统目录。
#                   必须支持 DESTDIR 变量。

# ----- 5.5 可选函数（钩子）-----
# prepare()         解压源码后、build() 之前调用，用于打补丁、修改构建文件等。

# =============================================================================
#  6. 环境变量（系统注入与配方可用的约定）
# =============================================================================
# DESTDIR           主脚本设置，仅 install() 内有效。
# PKG_CONFIG_LIBDIR 可由 config.sh 或配方设置，控制 pkg-config 搜索路径。
# PKG_CONFIG_SYSROOT_DIR  同上，pkg-config 的 sysroot。
# PATH              系统标准路径，构建工具链应在其中。
# CC, CXX           交叉编译时通常由工具链设置，配方一般不需修改。

# =============================================================================
#  7. 公共库提供的函数
# =============================================================================
# lib/common.sh:
#   info <msg>      绿色 [INFO] 输出
#   warn <msg>      黄色 [WARN] 输出
#   error <msg>     红色 [ERROR] 输出并退出脚本
#
# lib/fetch.sh:
#   download <url> <output_path>  下载文件到指定路径，支持 wget/curl，自动缓存
#
# lib/unpack.sh:
#   extract <archive> <dest_dir>  解压 .tar.gz/.tar.xz/.zip 等到目标目录

# =============================================================================
#  8. 变量继承与作用域示例 (以构建 flac 为例)
# =============================================================================
#  1. 用户执行 ./main/main.sh flac
#  2. 主脚本加载 lib/*.sh，再 source config.sh，此时 shell 中有 TARGET_HOST 等
#  3. 主脚本 source packages/flac/recipe.sh，获得 VERSION, SRC_URI
#  4. 主脚本执行 build_pkg flac：
#     - download "$SRC_URI" "$dl_path"
#     - extract "$dl_path" "$pkg_work/src"
#     - 进入 $src_dir
#     - 执行 prepare (如果存在)
#     - 执行 build -> 函数内可直接使用 TARGET_HOST, PREFIX 等
#     - 设置 DESTDIR="$dest_dir" 并导出，执行 install
#  5. 收集 $dest_dir 内容到 output/flac/

# =============================================================================
#  9. 扩展计划中的变量（预留）
# =============================================================================
# DEPENDS           声明当前包依赖的其他源码包名，实现自动依赖构建顺序
# CONFIGURE_EXTRA   数组，存放额外的 configure 参数，供 build() 拼接
# CCACHE_DIR        指定 ccache 缓存目录，加速重复构建
# BUILD_IN_CONTAINER 标记是否需要在容器/chroot 中构建

# =============================================================================
# 10. 开发约定与注意事项
# =============================================================================
#  - 配方不要写 #!/bin/bash 或 exit，它只是被 source 的片段。
#  - 配方中的变量名避免与主脚本内部变量冲突（如 pkg_name, recipe_dir 等）。
#  - 所有需要写入文件系统的操作必须指向 $DESTDIR，绝对不要直接安装到
#    $PREFIX 对应的系统路径。
#  - 如果配方需要补丁，应将补丁放在 packages/<包名>/patches/ 下，
#    并在 prepare() 中通过相对路径使用。
#  - 下载链接应尽可能使用 HTTPS，且版本号建议通过变量拼接，方便升级。
#  - 全局配置 config.sh 建议不纳入版本控制（加入 .gitignore），
#    每个开发者根据自己的环境创建。
